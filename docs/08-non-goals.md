# 08. Non-Goals

하지 **않을** 것들. 이 리스트는 범위 방어를 위한 공식 문서입니다.
이슈나 피드백으로 "이것도 해달라"가 들어왔을 때 여기서 판단 기준을 찾습니다.

## 영원히 하지 않을 것 (Never)

### ❌ 자체 메신저

Slack/Discord/카톡을 대체하려 하지 않습니다.

**이유**:
- 메신저는 네트워크 효과가 지배하는 시장. 기능이 아무리 좋아도 팀이 갈아타지 않음.
- 우리 제품의 가치 ("에이전트 작업 → 팀 자산")는 메신저 없어도 100% 달성 가능.
- 대신 Slack/Discord **봇**으로 연결. 링크 공유, 알림, 슬래시 커맨드.

**대응 원칙**: "메신저 기능 추가해달라"는 요청은 "봇 통합 기능 추가" 요청으로 리프레임.

### ❌ 범용 Wiki / Notion 대체

모든 문서 유형을 담는 범용 툴이 되지 않습니다.

**이유**:
- 우리는 **에이전트 작업의 산출물**에 최적화됨
- 범용으로 가는 순간 차별화 소실
- 회의록, 일기, 개인 메모, 디자인 스펙 같은 건 다른 도구에서

**대응 원칙**: 코딩 에이전트가 생성하는 artifact 타입만 지원. 그 외는 "Notion에서 쓰세요"라고 명시.

### ❌ 범용 프로젝트 관리 도구

Jira/Linear의 전체 스펙을 추구하지 않습니다.

**이유**:
- 스프린트 계획, 번다운 차트, 벨로시티, 워크로드 관리는 에이전트 워크플로우의 핵심이 아님
- 이 길로 가면 경쟁 제품들과 feature parity 경쟁 시작 → 필패
- Task는 artifact의 한 타입으로서, artifact-centric으로만 다룸

**대응 원칙**: "Gantt 차트 추가해달라" 같은 요청은 거절. 칸반 정도는 V1.x에 검토.

### ❌ LLM 자체 호스팅 / 모델 제공

Varn이 자체 LLM을 돌리거나 특정 모델에 종속되지 않습니다.

**이유**:
- LLM은 에이전트 클라이언트의 책임 (Claude Code가 Claude를 부르고, Cursor가 자체 모델을 부름)
- Varn이 LLM 돌리기 시작하면 비용/인프라 부담 폭증
- 모델 중립이어야 모든 에이전트를 품을 수 있음

**대응 원칙**: Varn 서버는 LLM을 직접 부르지 않음. 필요 시 옵션 기능으로 "외부 LLM API 연결" 정도만 제공.

### ❌ 코드 자동 생성 / 에이전트 실행

Varn 자체가 코딩 에이전트가 되지 않습니다.

**이유**:
- Cursor, Claude Code, Cline과 경쟁하는 순간 우리 가치 제안 붕괴
- 우리는 그들이 만든 출력을 다루는 **인프라 레이어**
- "Varn에서 직접 코드 수정" 기능은 범위 밖

**대응 원칙**: Varn 안에서 에이전트를 "실행"하는 건 없음. TC Runner(V1.1)는 예외적으로 자동 실행 에이전트 하나를 둠.

## V1에서 하지 않을 것 (V2+ 고려)

### 🔄 SSO / RBAC / 엔터프라이즈

**이유**: V1 타겟은 소규모 팀 (2–10명). 엔터프라이즈 기능은 V1에 필요 없음.
**고려 시점**: V2 이후. Open Core 전략 시 상용 tier.

### 🔄 멀티 테넌트

**이유**: V1은 self-host 1 팀 = 1 인스턴스. 
**고려 시점**: 클라우드 호스팅 시작 시 (V2).

### 🔄 모바일 전용 앱

**이유**: 코딩 에이전트 워크플로우는 데스크톱 중심.
**고려 시점**: 모바일 read-only 뷰어 수준으로 V2 이후.

### 🔄 플러그인 시스템

**이유**: 너무 일찍 열면 API 안정화 이전에 외부 의존성이 락인됨.
**고려 시점**: V1.x에서 내부 구조 안정화 후.

### 🔄 다국어 UI

**이유**: V1은 영어만. 한국어는 내부적으로 본인 사용으로만.
**고려 시점**: V1.x 이후. 커뮤니티 기여로 추가.

### 🔄 실시간 협업 (다중 커서 등)

**이유**: Artifact 편집은 주로 검수 작업이라 실시간 협업 불필요.
**고려 시점**: 명확한 사용자 니즈 있을 때만.

## 유사해 보이지만 우리가 아닌 것들

V1 출시 후 "그거랑 뭐가 다르냐"는 질문이 나올 것. 미리 정리:

### vs Notion / Confluence
- Notion: **사람이 작성, 사람이 독자**. Free-form.
- Varn: **에이전트가 작성, 사람이 검수, 사람+에이전트가 독자**. 타입 강제.

### vs Linear / Jira
- Linear: 태스크 관리 중심. 팀 워크플로우 최적화.
- Varn: Task는 artifact의 한 종류. 중심은 지식-태스크-코드의 결합.

### vs GitHub Issues / Discussions
- GitHub: 코드 저장소에 붙은 협업 도구. 에이전트 통합은 플러그인 수준.
- Varn: 에이전트 통합이 1급 시민. MCP가 표준 인터페이스.

### vs GitHub Copilot Workspace / Cursor Background
- 이들: 에이전트가 직접 코드 작성
- Varn: 그 에이전트들이 만든 **산출물의 저장/관리/재사용** 레이어

### vs cairn-dev/cairn
- cairn-dev: 백그라운드 에이전트 실행 시스템 (코드 작성 자동화)
- Varn: 에이전트 출력의 지식화 (코드 작성은 외부 에이전트가 함)

### vs MCP 서버들 (Notion MCP, Linear MCP)
- 기존 MCP: 기존 제품의 **연결자**. 그 제품의 제약/강점 그대로.
- Varn MCP: **첫날부터 MCP-native 설계**. Write 강제 게이트가 제품의 핵심.

## 거절의 기술 (이슈 응대 가이드)

GitHub Issue에 기능 추가 요청이 들어왔을 때의 판단 플로우:

```
1. 이것이 "에이전트 → 팀 자산" 루프를 강화하는가?
   NO → 거절 (non-goals에 해당)
   YES → 다음
   
2. 이것이 이미 다른 도구로 해결되는가?
   YES (예: 메신저는 Slack) → 연결만 제공, 자체 구현 X
   NO → 다음
   
3. 이것이 최소 3명 이상 요청했는가?
   NO → 🔖 백로그, 데이터 누적
   YES → 다음
   
4. 이것이 V1 flagship feature의 성숙을 늦추는가?
   YES → V1.x 이후
   NO → V1 검토
```

## 마무리

이 문서가 지켜져야 Varn이 **선명한 제품**으로 살아남습니다.
Feature creep은 1인/소수 OSS의 최대 적입니다.
"할 수 있는 것"과 "해야 하는 것"을 구분하는 것이 메인테이너의 일입니다.
