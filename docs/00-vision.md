# 00. Vision

> **북극성 한 문장:** 에이전트의 작업이 개인과 팀의 기억이 되는 곳.
> *Where agent work becomes lasting memory.*

## 제품 정의 (한 문장)

> **사람은 타이핑하지 않는 위키.**
> 모든 쓰기는 에이전트를 통해서만 일어나고, 사람은 승인·거절·방향 제시만 한다.

## 왜 지금인가

2026년, 코딩 에이전트는 개인 생산성을 10배 끌어올렸습니다. 하지만 **지식의 생산성**은 아직 그 레버리지를 흡수하지 못하고 있습니다. 이유는 단순합니다 — 에이전트의 출력이 영속 자산으로 변환되는 파이프라인이 없습니다.

현재 개발자의 현실:

- 디버깅 2시간 → 세션 닫으면 증발 → 일주일 뒤 같은 문제 또 2시간
- "그때 그 대화 어디 있었지" → 월 단위로 쌓인 세션·채널·채팅 뒤지기 → 대부분 포기
- 레거시 분석을 에이전트로 정리 → Notion/Confluence에 수동 복붙 → 포맷 제각각 → 검색 안 됨
- 팀원이 본 문서의 반은 이미 stale, 본인은 모름
- 애자일하게 일할수록 문서와 현실의 거리가 벌어짐

**과도기의 수동 해결책들이 있습니다.** 본인처럼 "wiki.js + OpenProject + 수동 스킬"로 개인과 팀의 번역가 역할을 하는 사람이 각 팀마다 한 명씩 나타나고 있습니다. 스케일하지 않고, 그 개인이 없으면 무너집니다.

Varn은 이 역할을 **제품화**합니다.

## 디자인 원칙 (우선순위 순)

### 원칙 1. Human-writable surface is zero.

**사람은 위키에 직접 타이핑하지 않는다.** 오탈자 교정, 링크 수정, 이미지 교체까지 **전부 에이전트와의 대화를 통해서만** 일어난다. UI에는 편집 버튼이 없다.

이유:
- 대 에이전트 시대에 사람의 흐물흐물하고 구체적이지 않은 파편들이 문서에 그대로 스며들면 **곧 노이즈**
- 사람의 역할은 판단·승인·거절·방향 제시. 기계적 정리는 에이전트의 일
- UI에 편집 버튼이 있는 순간 타협이 시작되고, 타협이 시작되면 Notion과 같아짐

이 원칙이 Varn을 **기존의 모든 wiki/task 제품과 근본적으로 다르게** 만든다. Notion/Confluence/Wiki.js는 자기 사용자 기반 때문에 절대 받아들일 수 없는 선언이고, 이것이 **구조적 해자**다.

### 원칙 2. Promote는 에이전트가 제안하고 사람이 승인한다.

기존 툴들의 중심 동사는 `Create`입니다 (Notion, Linear). Varn의 중심 동사는 **`Promote`** — 에이전트가 세션 중 "체크포인트"로 판단되는 시점에 사용자에게 **역으로** "이거 정리할까요?" 를 제안하고, 사용자는 OK(또는 "아직 아님")만 하면 되는 승격 행위.

"쌓인 출력 중에서 무엇을 건질까"의 세계관. 시그널/노이즈 비율이 역전된 시대의 중심 동사입니다.

**꼭 완결된 정보만 남길 필요는 없습니다.** 유의미하게 정리된 사안이면 `partial` 상태로 일단 기록하고 본다. Artifact는 `draft → partial → settled`로 성숙해진다.

### 원칙 3. 문서와 Tool은 active agent다.

기존 wiki는 문서를 **수동적 객체**로 취급합니다. 외부에서 CRUD를 때리는 대상.

Varn의 문서와 tool은 **능동적 주체**입니다. 에이전트가 뭔가를 쓰려 할 때 Tool이 **"그 전에 이거부터 탐색했냐, 기존에 비슷한 거 없는지 봤냐, 필수 필드 다 채웠냐"** 를 역으로 되묻습니다. MCP가 에이전트에게 단순 응답하는 게 아니라 **에이전트에게 더 많은 작업을 시킵니다**.

### 원칙 4. 제약이 곧 가치다.

Notion은 "뭐든 써라" 제품입니다. Varn은 **"이 규칙 지켜야 써라"** 제품입니다.

에이전트 write volume이 사람의 100배인 시대에 필요한 건 자유가 아니라 **규율**입니다. 제약 없는 wiki는 쓰레기장이 됩니다.

### 원칙 5. 코드와 문서는 결합되어야 한다.

모든 artifact는 커밋 해시, PR 번호, 파일 경로에 **고정(pin)** 됩니다. 코드가 변하면 artifact는 stale 표시. 이것이 문서와 현실의 드리프트를 구조적으로 차단합니다.

## 타겟 사용자

**V1 타겟: 코딩 에이전트를 이미 쓰는 솔로 개발자 ~ 10인 소규모 팀.**

구체적으로:
- 최소 한 명이 Claude Code / Cursor / Cline / Codex 등 코딩 에이전트를 일상 사용
- 세션·채팅이 월 단위로 누적되어 "그때 그거" 탐색이 고통스러운 상태
- (팀의 경우) wiki나 이슈 트래커 조합으로 협업 중 (Notion+Linear, Confluence+Jira 등)
- 변경이 잦은 애자일 워크플로우
- 도구 선택 자율성이 있음

### 1급 시민으로서의 솔로 개발자

**솔로도 Varn의 가치 절반은 그대로 누립니다.** 내 세션 기억의 인덱싱 + 반-자동 구조화만으로도 "월 단위 채팅 검색 지옥"이 사라집니다. 팀 공유 레이어는 그 위에 얹힙니다 — 있으면 좋고, 없어도 동작.

### 자율 에이전트 환경 (OpenClaw 등)

human-out-of-the-loop 환경에서도 동작합니다. 에이전트가 체크포인트 판단·기록을 자율로 수행하고, 사용자·관리자는 정리된 위키와 태스크만 보고 프로젝트 상태를 파악. `VARN.md`의 mode를 `auto`로 두면 이 시나리오가 성립합니다.

### V1 제외

- 대기업 엔터프라이즈 (SSO, 감사로그 등 별도)
- 비개발 지식노동자 팀
- 코딩 에이전트를 전혀 쓰지 않는 사용자 (Varn은 agent-only write이므로 에이전트 없이는 read-only)

## 유즈케이스 (V1에서 상정)

1. **솔로 "세션 아카이브"** — 개인이 Claude Code로 일상 작업. 유의미한 사안마다 에이전트가 역으로 "정리할까요?" → OK → 반-wiki에 누적. 나중에 URL 던져서 대화 재개.
2. **신규 프로젝트 부트스트랩** — "설계 문서 정리해줘" → 에이전트가 Vision/Problem/Concepts 스켈레톤을 Varn typed schema에 맞춰 draft → 사람과 토의하며 채워짐. (이 리포지토리 자체가 1호 사례.)
3. **레거시 프로젝트 역공학** — 에이전트가 기존 repo를 스캔 → 자동 생성된 Feature/Flow 스켈레톤 제안 → 사용자가 OK/수정하며 팀 지식 부트스트랩.
4. **팀 협업 레이어** — 여러 에이전트 사용자가 같은 인스턴스에 붙어 각자 promote. 서로의 artifact에 중복·충돌이 자동 감지되고, 승인은 도메인 owner가.

## 성공의 정의

**단기 (OSS 관점)**
- GitHub star 5,000+
- Hacker News 1면 진입
- 솔로·팀 합쳐 최소 5 인스턴스 30일 이상 지속 가동

**중기 (사용성 관점)**
- 솔로 사용자 10+ 명이 매일 사용
- 팀 3+ 개가 3개월 지속 사용
- 사용자당 artifact 누적 100+ 개
- "Varn 없던 때로 돌아갈 수 없다"는 feedback

**장기 (생태계 관점)**
- Claude Code, Cursor, Cline, Codex 등 주요 에이전트의 공식 integration
- "Varn memory"가 개인·팀 자산으로 인수·합병 시 넘어가는 지식 베이스

## 비성공의 정의

- **Human-writable surface이 허용됨** — 편집 버튼 타협. 제품 정체성 붕괴.
- "또 하나의 Notion-like 제품" 포지셔닝으로 인식됨
- 에이전트 write-path 강제가 선택 기능으로 격하됨
- 범용 협업 플랫폼으로 스코프 확장 (메신저, 캘린더, CRM 등)
- 엔터프라이즈 기능 (SSO, RBAC, 감사로그)이 V1 우선순위를 침범
