# 01. Problem

Varn이 풀려는 구체적인 실패 모드들입니다. 이 문서는 실제 현장 경험에서 역산된 것이며, 해결 메커니즘은 [05 Mechanisms](05-mechanisms.md)에 정의합니다.

## 실패 모드 6가지

### F1. 중복 생성 (Duplicate Creation)

**상황**: 결제 플로우 문서가 이미 있음. 변경 사항이 생겨서 에이전트에게 "이번 변경 내용 정리해줘"라고 요청하면, 에이전트는 **기존 문서를 모른 채** 새 문서를 만든다.

**원인**: 기존 wiki 도구는 에이전트의 write 요청을 **컨텍스트 없이** 받아들인다. "찾아보고 있으면 업데이트하고 없으면 새로 만들어"라는 지시를 매번 사람이 내려야 함.

**증상**:
- 같은 주제의 문서가 여러 버전으로 공존
- 문서 수가 수백 개를 넘기면 검색이 무의미해짐
- 팀원이 어느 문서가 "진짜"인지 모름

**뼈아픔 등급**: ★★★★★

---

### F2. 경계 침범 (Scope Bleeding)

**상황**: B 피쳐 논의 중 A 피쳐까지 건드리게 됨. 에이전트에게 정리를 요청하면 "BA 혼합 문서"가 만들어지거나, B 문서에 A 수정 사항이 섞여 들어감.

**원인**: 문서의 경계(scope)가 명시적으로 선언되지 않음. 에이전트는 대화 맥락만 보고 판단.

**증상**:
- A 문서는 그대로 stale, B 문서에 A 변경사항 숨어있음
- 나중에 A를 찾는 사람은 B 문서를 못 찾음
- "이건 어디 문서에 있지?" 탐색 비용 증가

**뼈아픔 등급**: ★★★★

---

### F3. 포맷 드리프트 (Format Drift)

**상황**: 분석 문서, 기획 문서, ADR(아키텍처 결정 기록), 디버깅 로그 — 각 종류마다 적법한 포맷이 있음. 하지만 세션마다 에이전트가 다른 포맷으로 씀. 때로는 포맷이 섞임.

**원인**: 포맷이 **관습**으로만 존재하고, **스키마**로 강제되지 않음. 에이전트에게 "이 포맷으로 써줘"를 매번 프롬프팅해야 함.

**증상**:
- 같은 종류의 문서인데 구조가 다 다름
- "ADR인데 Consequences 섹션이 비어있음" 같은 케이스
- 팀원이 문서 열 때마다 구조 파악 비용

**뼈아픔 등급**: ★★★

---

### F4. TC 관리 공백 (Test Case Gap)

**상황**: OpenProject/Jira는 TC를 1급으로 지원하지 않음. 노트 섹션에 텍스트 덩어리로 TC를 적어두게 됨. AI로 자동화 가능한 TC와 사람이 E2E로 해야 하는 TC의 구분도 수동.

**원인**: 태스크 매니저가 "작업 관리"만 상정하고 "검증 관리"를 외면. TC는 보조 메모 수준.

**증상**:
- TC가 구조화되지 않아 실행 여부 추적 불가
- "이 feature 닫아도 되나?"에 대한 기계적 답이 없음
- AI로 자동 실행 가능한 TC들이 수동 검토에 머뭄

**뼈아픔 등급**: ★★★★

---

### F5. 저해상도 추적 (Low-Fidelity Tracking)

**상황**: 애자일하게 변경이 잦음. 코드가 매일 바뀌는데 wiki/task는 못 따라옴. 결과적으로 "머리만 최신이고 꼬리는 stale"인 상태가 누적.

**원인**: 문서와 코드 간의 **의존성 그래프**가 없음. 코드가 변했을 때 어떤 문서가 낡았는지 알 방법이 없음.

**증상**:
- 문서를 신뢰할 수 없음 → 팀원이 문서 대신 코드를 다시 읽음 → 문서 존재 이유 소멸
- "이 문서 언제 적 거야?" 묻는 횟수 증가
- 에이전트가 stale 문서를 참조해 잘못된 분석 생성

**뼈아픔 등급**: ★★★★★

---

### F6. 세션 검색 지옥 (Session Retrieval Hell) ← NEW

**상황**: 지난 3주 사이 어느 세션에선가 "그 PG사 타임아웃 건" 맥락을 깊게 다뤘음. 지금 다시 그 맥락이 필요한데 **어디에 있었는지 못 찾음**. Claude Code / Cursor / ChatGPT / Codex 각각의 세션 리스트를 뒤지는데 검색이 개차반이라 사실상 새로 대화 시작.

**원인**:
- 에이전트 IDE의 세션 리스트는 대화가 월 단위로 쌓이는 속도를 상정하지 않은 UX
- 제목이 자동 생성되지만 의미있는 신호가 없음
- 본문 검색은 있어도 의미 검색·타입 필터·시간 범위 조합이 빈약
- 여러 에이전트를 병행하면 이 지옥이 **N배로** 누적

**증상**:
- 같은 맥락을 N번째 재구축 → 개인 생산성이 오히려 후퇴
- 중요한 결론이 "어딘가에 있었는데"로 남고 실제로는 소실
- 에이전트 전환 시 그동안의 맥락이 통째로 버려짐

**뼈아픔 등급**: ★★★★★

**누가 이걸 제일 크게 앓나**: 솔로 개발자. 팀 위키·태스크가 없어서 세션 로그가 유일한 기억 소스인 사용자. V1이 솔로를 1급으로 품는 이유.

---

## 실패 모드 간 관계

6가지는 독립적이지 않고, 두 개의 근본 원인에서 나옵니다:

> **(A) 문서가 passive object으로 취급된다.**
> 외부 행위자(사람/에이전트)가 "알아서 경계를 지키고, 중복을 피하고, 포맷을 맞추고, 전파를 감지해야" 함. 그 책임이 외부 컨텍스트에 있으니 반드시 무너진다. → F1~F5

> **(B) 에이전트 세션이 1회성 휘발 자원으로 취급된다.**
> 세션을 닫으면 그 안의 시그널이 사라지고, 설사 저장되어 있어도 검색 기능이 받쳐주지 않는다. → F6

해결 방향:

> **(A) 문서를 active agent로 만든다.** 들어오는 write 요청을 문서가 스스로 심사한다.
> **(B) 세션을 영속 자원으로 다루고, Promote를 통해 승격 경로를 연다.** 유의미하면 일단 `partial`로 기록.

구체적 메커니즘은 [05 Mechanisms](05-mechanisms.md).

## 우선순위 — V1 Flagship

| 실패 모드 | 뼈아픔 | 해결 난이도 | V1 포함? |
|----------|--------|-----------|----------|
| F1. 중복 생성 | ★★★★★ | 중 | ✅ Flagship |
| F2. 경계 침범 | ★★★★ | 중 | ✅ |
| F3. 포맷 드리프트 | ★★★ | 낮음 | ✅ |
| F4. TC 관리 공백 | ★★★★ | 높음 | 🔄 V1 기본 + V1.1 Runner |
| F5. 저해상도 추적 | ★★★★★ | 높음 | ✅ 기본 pin, V1.1 대시보드 |
| F6. 세션 검색 지옥 | ★★★★★ | 중 | ✅ Flagship (솔로 엔트리) |

**V1 flagship은 F1 + F3 + F6 세 개 묶음**:
- F1(중복 생성)은 팀·솔로 모두 가장 체감 큼
- F3(포맷)은 난이도 낮고 함께 해결 시 임팩트 큼
- F6(세션 검색 지옥)은 솔로 사용자의 entry pain이자 초기 star 확보 동력
- F2(경계)는 F1 해결 메커니즘의 부산물로 자연 해결
- F4(TC)와 F5(전파)는 기반만 V1에, 풍부한 UX는 V1.1

※ 이 우선순위는 논의 가능. [07 Roadmap](07-roadmap.md) 참조.

## 현장 일화 (참고)

설계팀이 참고할 실제 관찰 사례:

> "결제플로우에 대해서 문서를 작성했고 팀원간에 공유됐고, 파악하고있었는데, 다시 결제플로우의 변동이 생겼는데 일단 급하니까 변동건에 대해 논의나누고 문서화 요청하면, 내가 굳이 '관련문서 있으면 찾아보고'라고 말 안 하면 문서가 새로 생기는 경우도 있고..." — F1

> "분명히 다른 피쳐로 대화가 오갔던건데 결국 경계를 넘어서 B를 논하다가 A까지 바뀌게 되는 상황에서 이러면 A수정 B수정 이렇게 됐어야 될텐데 A를 그대로 두고 BA가 생성되는 케이스..." — F2

> "포맷도 제각각, 사실 특정 사안마다 적법한 포맷들이 있어서 흔들릴 순 있는데, 상당히 무너지는 경우가 많음." — F3

> "변경이 잦은데 이걸 wiki와 op가 상시 고해상도로 따라오긴 힘들었고 머리만 쫒아오는 식으로 희미한 꼬리가 증분되는 형태로 자주 빠졌음." — F5

> "검색도 개차반임 진짜 월단위로 채널이 늘어나면 찾기 진짜 힘듬. 매번 본인의 에이전트와의 대화세션을 뒤져가며 '그때 이맥락 다시보고싶은데 어딨지' 하면서..." — F6

이 관찰들이 F1–F6의 원천입니다.
