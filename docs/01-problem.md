# 01. Problem

Pindoc이 풀려는 구체적인 실패 모드들. 해결 메커니즘은 [05 Mechanisms](05-mechanisms.md)에 정의합니다.

## 실패 모드 6가지

### F1. 중복 생성 (Duplicate Creation)

**상황**: 결제 플로우 문서가 이미 있음. 변경이 생겨 에이전트에게 "정리해줘"라고 하면, 에이전트는 **기존 문서를 모른 채** 새 문서를 만든다.

**원인**: 기존 wiki는 에이전트 write를 **컨텍스트 없이** 받아들인다.

**증상**: 같은 주제의 문서 여러 버전 공존, 수백 개 넘으면 검색 무의미, 팀원이 어느 게 "진짜"인지 모름.

**뼈아픔**: ★★★★★

---

### F2. 경계 침범 (Scope Bleeding)

**상황**: B 피쳐 논의 중 A 피쳐까지 건드림. "BA 혼합 문서" 또는 B 문서에 A 수정이 숨어 들어감.

**원인**: 문서 경계가 명시 선언되지 않음.

**증상**: A 문서는 stale, B 문서에 A 변경이 숨어있음, "어디 문서에 있지?" 탐색 비용.

**뼈아픔**: ★★★★

---

### F3. 포맷 드리프트 (Format Drift)

**상황**: 분석·기획·ADR·디버깅 각 종류마다 적법한 포맷이 있는데 세션마다 제각각.

**원인**: 포맷이 **관습**으로만 존재, **스키마**로 강제되지 않음.

**증상**: "ADR인데 Consequences 비어있음", 팀원이 문서 열 때마다 구조 파악 비용.

**뼈아픔**: ★★★

---

### F4. TC 관리 공백 (Test Case Gap)

**상황**: OpenProject/Jira는 TC 1급 아님. 노트 섹션에 텍스트 덩어리. AI 가능 TC와 사람 E2E TC 구분도 수동.

**원인**: 태스크 매니저가 "작업 관리"만, "검증 관리" 외면.

**증상**: 실행 여부 추적 불가, "Feature 닫아도 되나?"에 기계적 답 없음.

**뼈아픔**: ★★★★

---

### F5. 저해상도 추적 (Low-Fidelity Tracking)

**상황**: 코드가 매일 바뀌는데 wiki/task는 못 따라옴. "머리만 최신이고 꼬리는 stale".

**원인**: 문서와 코드 간의 **의존성 그래프** 없음.

**증상**: 문서 신뢰 불가 → 코드 다시 읽음 → 문서 존재 이유 소멸, "이 문서 언제 적었지?"

**뼈아픔**: ★★★★★

---

### F6. 과거 맥락 재발견 지옥 (Context Retrieval Hell)

**상황**: 지난 3주 중 어느 세션에선가 "PG사 타임아웃" 맥락을 깊게 다룸. 지금 그 맥락이 필요한데 **어디서도 찾을 수 없음**.

**원인 두 갈래**:
1. **정리 안 됨** — 세션 원본만 있고 artifact로 promote 안 됨 → 찾기 불가
2. **검색 약함** — promote는 됐는데 Notion/wiki.js 검색으로는 한국어↔영어 네이밍 gap, 추상 개념 매칭 실패

**증상**:
- 같은 맥락 N번째 재구축 → 개인 생산성 오히려 후퇴
- 중요한 결론이 "어딘가 있었는데"로 남고 소실
- 에이전트 전환 시(Claude Code→Cursor) 맥락 통째로 버려짐

**뼈아픔**: ★★★★★

**누가 제일 앓나**: Solo 개발자. 팀 위키·태스크가 없어 세션 로그가 유일한 기억 소스.

---

## F6의 Scope — Pindoc이 풀고/안 푸는 것

**중요 경계**: F6의 원인 중 **(2) 검색 약함**은 Pindoc이 직접 푼다. **(1) 정리 안 됨**은 Pindoc이 **Promote 문화로** 유도해서 간접 해결.

**Pindoc이 안 하는 것**:
- **에이전트 클라이언트의 세션 원본 파일(JSONL, SQLite 등)을 흡수하지 않는다.** Claude Code / Cursor / Cline / Codex 각 로컬 파일은 각 앱의 책임.
- "너절한 raw 채팅 로그"는 해당 앱(Claude 앱, Codex 앱)에서 보는 거지 Pindoc 관심사 아님.
- Pindoc이 관리하는 건 **에이전트가 정제해 promote한 artifact** 만.

**Pindoc이 하는 것** (F6 해결 경로):
- 정제된 artifact에 **의미 검색 + Area/Type/Completeness 필터 + Cmd+K** 조합
- 한국어↔영어 네이밍 gap을 **pgvector 임베딩**으로 해소
- Artifact가 `source_session: SessionRef` 메타를 담아 "이건 어느 세션에서 나온 것"을 표시 — 원하면 사용자가 해당 클라이언트에서 원본을 열어봄
- **Promote 문화 유도**로 "정리 안 됨"의 빈도 자체를 낮춤 (PINDOC.md `mode: auto`)

이게 정직한 포지셔닝: **"정리된 것 내에서 찾기 보장, 정리 안 된 것은 각 앱 책임"**.

## 실패 모드 간 관계

6가지는 두 개의 근본 원인에서 나옵니다:

> **(A) 문서가 passive object으로 취급된다.**
> 외부 행위자가 "경계·중복·포맷·전파를 다 알아서 해야" 함. 책임이 외부에 있으니 반드시 무너진다. → F1~F5

> **(B) 에이전트 세션 시그널이 promote 경로 없이 방치된다.**
> raw 세션이 영속 자산으로 승격될 파이프라인이 없다. → F6

해결 방향:
> **(A) 문서를 active agent로 만든다.** Write 요청을 문서·Tool이 스스로 심사.
> **(B) Promote를 중심 동사로 박는다.** 유의미하면 `partial`로라도 기록하고, 검색은 artifact 대상으로.

구체 메커니즘은 [05 Mechanisms](05-mechanisms.md).

## 우선순위 — V1 Flagship

| 실패 모드 | 뼈아픔 | 해결 난이도 | V1 포함? |
|----------|--------|-----------|----------|
| F1. 중복 생성 | ★★★★★ | 중 | ✅ Flagship |
| F2. 경계 침범 | ★★★★ | 중 | ✅ |
| F3. 포맷 드리프트 | ★★★ | 낮음 | ✅ |
| F4. TC 관리 | ★★★★ | 높음 | 🔄 V1 기본 + V1.1 Runner |
| F5. 저해상도 | ★★★★★ | 높음 | ✅ 기본 pin, V1.1 대시보드 |
| F6. 과거 맥락 재발견 | ★★★★★ | 중 | ✅ Flagship |

**V1 flagship은 F1 + F3 + F6 세 개 묶음** — F1은 모두 체감 큼, F3는 난이도 낮고 임팩트 큼, F6은 Solo entry pain이자 초기 star 확보 동력. F2는 F1 부산물로 자연 해결.
