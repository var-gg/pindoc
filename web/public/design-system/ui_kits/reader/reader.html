Session store migration — Redis to Postgres

live ADR · 0042 platform/auth ·
ccwritten by claude-code
· promoted 2h ago · verified against a3f2c1b

Context

We run session state in acme-platform through a Redis cluster fronted by the auth-gateway service. Over the last quarter we saw three incidents where a cold replica promotion caused session loss for ~12 minutes — a worse outcome than any plausible p99 latency penalty.

This ADR proposes migrating session storage to the existing Postgres cluster and retiring the Redis dependency for auth.

Decision

Use Postgres as the canonical session store. A write-through LRU in-process cache keeps hot-path p50 under 2ms. The existing oauth-device-flow implementation requires no changes.

Sessions are not hot data. They are medium-warm data we were treating as hot because Redis made it easy.

Trade-offs accepted

  • p99 latency rises from ~4ms to ~11ms on cache miss. Acceptable given the traffic profile.
  • Postgres connection pool pressure increases ~8%. pgbouncer-sizing already has headroom.
  • Loses Redis-side TTL automation; replaced by a nightly janitor job (session-janitor-runbook).

Rollout

Dual-write for 14 days, read from Postgres, fall back to Redis on miss. After burn-in, flip SESSION_STORE=pg and decommission the Redis cluster.
